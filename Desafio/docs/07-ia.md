# 07 - IA

## Objetivo

A IA deve deixar o TeachGram Pro mais chamativo para portfolio sem transformar o app em refem de uma API externa. Ela deve ajudar o usuario a criar conteudo, moderar publicacoes e demonstrar arquitetura plugavel.

Provider principal escolhido:

- Groq Cloud API, usando `groq/compound` para geracao e `groq/compound-mini` para respostas rapidas, hashtags e moderacao leve.

Regra central:

- A rede social deve funcionar mesmo sem IA.

## Fontes Oficiais

- [Groq Compound](https://console.groq.com/docs/compound)
- [Groq OpenAI compatibility](https://console.groq.com/docs/openai)

Observacao:

- A Groq documenta a API como majoritariamente compativel com OpenAI, usando `https://api.groq.com/openai/v1` como base URL.
- `groq/compound` pode usar ferramentas internas como busca web e execucao de codigo. Por isso, a aplicacao deve tratar IA como recurso externo com cota, timeout, rate limit e fallback.

## Casos De Uso

### Gerar Legenda

Entrada:

- Tema.
- Tom.
- Idioma.
- Descricao opcional da imagem.

Saida:

- 1 a 3 sugestoes curtas.
- Texto editavel antes de publicar.

Prompt conceitual:

```text
Voce ajuda usuarios de uma rede social chamada TeachGram.
Gere legendas curtas em pt-BR.
Nao use texto ofensivo.
Nao invente dados pessoais.
Retorne apenas JSON com a chave suggestions.

Tema: {topic}
Tom: {tone}
Limite: 200 caracteres por sugestao.
```

### Sugerir Hashtags

Entrada:

- Titulo.
- Descricao.

Saida:

- 3 a 8 hashtags.
- Sem duplicatas.
- Sem hashtags ofensivas.

### Melhorar Bio

Entrada:

- Palavras-chave.
- Tom.
- Bio atual opcional.

Saida:

- 1 a 3 bios de ate 160 ou 300 caracteres, conforme UI.

### Moderar Conteudo

Entrada:

- Texto de post/comentario/perfil.
- URL de imagem opcional se provider suportar.

Saida:

- `SAFE`
- `UNSAFE`
- `REVIEW_REQUIRED`
- Categorias.

Regras:

- Conteudo `SAFE` pode ser publicado.
- Conteudo `UNSAFE` deve ser ocultado ou bloqueado.
- `REVIEW_REQUIRED` pode publicar oculto ate revisao ou avisar usuario.
- Falha tecnica vira `REVIEW_REQUIRED` se o conteudo parecer sensivel, ou deixa publicar com auditoria se for conteudo comum.

### Busca Semantica

Entrada:

- Texto de busca.

Saida:

- Posts/perfis relacionados semanticamente.

Recomendacao:

- Implementar depois da busca textual.
- Pode usar embeddings se houver provider gratuito disponivel.
- Se nao houver provider, manter endpoint como recurso opcional.

## Arquitetura Do Modulo AI

Interfaces:

```java
public interface AiProvider {
    CaptionResult generateCaption(CaptionRequest request);
    HashtagResult suggestHashtags(HashtagRequest request);
    BioResult improveBio(BioRequest request);
    ModerationResult moderate(ModerationRequest request);
}
```

Implementacoes:

- `GroqCompoundAiProvider`
- `FallbackAiProvider`
- `CompositeAiProvider`

`CompositeAiProvider`:

1. Tenta provider principal.
2. Aplica timeout.
3. Se falhar, registra `ai_jobs`.
4. Retorna fallback seguro.

## Configuracao

Variaveis:

```text
AI_ENABLED=true
AI_PROVIDER=groq
GROQ_API_KEY=...
GROQ_BASE_URL=https://api.groq.com/openai/v1
GROQ_TEXT_MODEL=groq/compound
GROQ_FAST_MODEL=groq/compound-mini
GROQ_MODEL_VERSION=
AI_TIMEOUT_MS=8000
AI_DAILY_LIMIT_PER_USER=30
AI_FALLBACK_ENABLED=true
```

Ambientes:

- `local`: provider real se chave existir; fallback se nao existir.
- `test`: provider fake.
- `demo`: provider real com rate limit agressivo e fallback.

## Rate Limit

Endpoints de IA devem ter limite proprio.

Sugestao:

- Caption: 10 por hora por usuario.
- Hashtags: 20 por hora por usuario.
- Bio: 10 por hora por usuario.
- Moderacao interna: limite por fila/job, nao por UI.

Resposta em limite:

```json
{
  "type": "https://teachgram.pro/problems/rate-limit",
  "title": "Too many requests",
  "status": 429,
  "detail": "AI daily limit reached. Try again later."
}
```

Frontend deve:

- Mostrar aviso curto.
- Permitir continuar manualmente.

## Persistencia

Use `ai_jobs` para:

- Auditoria.
- Debug.
- Mostrar maturidade tecnica.
- Medir latencia e falhas.

Nao salvar:

- API key.
- Prompt completo com dados sensiveis.
- Resposta bruta se puder conter informacao sensivel.

Salvar:

- Tipo do job.
- Provider.
- Modelo.
- Status.
- Latencia.
- Hash da entrada.
- Preview seguro da saida.
- Codigo de erro.

## Moderacao E Produto

Moderacao nao deve parecer censura misteriosa no produto. Mensagem recomendada:

```text
Nao conseguimos publicar esse conteudo agora. Revise o texto e tente novamente.
```

Para conteudo em revisao:

```text
Seu post foi salvo, mas ainda nao esta publico porque precisa de uma verificacao automatica.
```

## Fallbacks

### Caption Fallback

Entrada minima:

- Se houver titulo: "Compartilhando: {title}"
- Se nao houver: "Compartilhando um novo momento no TeachGram."

### Hashtag Fallback

Gerar hashtags simples por palavras-chave:

- Remover stopwords.
- Capitalizar palavras.
- Limitar a 5.

Exemplo:

```text
"Projeto fullstack com Spring e React" -> ["#Projeto", "#Fullstack", "#Spring", "#React"]
```

### Bio Fallback

```text
Criando, aprendendo e compartilhando projetos no TeachGram.
```

### Moderacao Fallback

Se provider falhar:

- Conteudo comum: permitir publicacao e registrar `ERROR`.
- Conteudo com termos localmente suspeitos: marcar `REVIEW_REQUIRED`.

## Prompts

Regras de prompt:

- Pedir JSON estruturado.
- Definir idioma.
- Definir limite de caracteres.
- Nao pedir pensamento passo a passo.
- Nao deixar modelo publicar automaticamente.
- Sanitizar entrada.
- Tratar output invalido.

Formato de resposta esperado para captions:

```json
{
  "suggestions": [
    "texto 1",
    "texto 2"
  ]
}
```

Se o modelo retornar texto invalido:

- Tentar parse robusto.
- Se falhar, usar fallback.
- Registrar erro `INVALID_MODEL_OUTPUT`.

## Seguranca Em IA

Riscos:

- Prompt injection.
- Conteudo ofensivo.
- Vazamento de dados.
- Custos por abuso.
- Dependencia de provider.

Mitigacoes:

- Rate limit.
- Timeout.
- Fallback.
- Nao enviar senha, token, email ou telefone para IA.
- Moderar entradas e saidas quando possivel.
- Logar apenas hash/preview seguro.
- CORS e auth nos endpoints.

## UX Da IA

Principios:

- IA sugere, usuario decide.
- Botao claro: "Sugerir legenda", "Gerar hashtags", "Melhorar bio".
- Mostrar "Gerado por IA" discretamente.
- Permitir editar antes de aplicar.
- Se falhar, texto manual continua funcionando.

Estados UI:

- Idle.
- Loading com skeleton ou spinner pequeno.
- Success com sugestoes.
- Fallback com aviso.
- Rate limited.
- Provider indisponivel.

## Testes

Unitarios:

- `CompositeAiProvider` chama fallback quando provider falha.
- Prompt builder respeita limites.
- Parser rejeita output invalido e usa fallback.
- Rate limit bloqueia excesso.

Integracao:

- Endpoint `/ai/caption` retorna provider real ou fallback.
- Endpoint `/ai/moderate` salva `moderation_events`.
- Falha da Groq Compound nao retorna 500 para caption.

E2E:

- Usuario abre criar post.
- Clica gerar legenda.
- Aplica sugestao.
- Publica post.
- Em modo provider fake falhando, fallback aparece e post manual funciona.

## O Que Dizer Em Entrevista

Ponto forte:

> Eu nao acoplei o produto a um modelo especifico. Criei um modulo de IA com provider plugavel, rate limit, timeout, persistencia de jobs e fallback. Assim a demo continua funcionando mesmo se a cota gratuita acabar.

Trade-off honesto:

> Como o projeto usa free tier, eu trato IA como enhancement. Em producao real, eu configuraria budget, observabilidade de custo, avaliacao de qualidade, moderacao mais robusta e possivelmente uma fila dedicada para jobs.
